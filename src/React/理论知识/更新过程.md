# React 的更新过程

React 的更新过程可以概括为：

**触发更新 → 创建 Update → 调度 → render 阶段计算变更 → commit 阶段提交变更**

## 1. 触发更新

更新的来源主要有三类：

- 初始化挂载：比如 `createRoot().render()`
- 状态更新: 比如 `setState`、Hook 的 `setState`
- 外部触发：`props` 变化、`context` 变化等

这些操作最终都会转化为：**给某个 Fiber 创建一个 Update，并从这个 Fiber 开始调度。**

所有更新最终都会走到 `scheduleUpdateOnFiber` 这个统一入口。

## 2. 创建 Update 并进入更新队列

调用 `setState` 后，React 不会立刻更新 DOM，而是先创建一个 `Update` 对象，里面记录本次更新的数据和优先级，然后放入对应 Fiber 的更新队列里：

```typescript
const update = {
  lane, // 本次更新的优先级
  action, // 新的 state 或 updater 函数
  next: null, // 链表指针，指向下一个 update
  // ...
};
```

- 类组件：Update 挂到 `fiber.updateQueue.shared.pending`（环形链表）
- 函数组件 Hook：Update 挂到 `hook.queue.pending`（同样是环形链表）

多个 setState 调用会产生多个 Update，它们串成链表，在 render 阶段才被统一消费。

## 3. 调度

React 18 用 **Lanes** 表示更新优先级。  
当某个 Fiber 产生更新时，会为它分配一个 `lane`，并把优先级信息沿父链向上冒泡到根节点。

然后 React 会根据优先级决定：

- 是同步执行还是异步执行
- 是否和其他更新合并
- 是否打断低优先级任务

批量更新的本质：在同一个事件回调或微任务中的多次 setState，共享同一个 lane，只会触发一次调度，在一次 render 中统一消费所有 Update。

## 4. render 阶段

- 可中断
- 可恢复
- 可被高优先级任务打断

主要工作：

1. 从根节点开始深度优先遍历 Fiber 树
2. beginWork：向下遍历，根据新旧 props/state 对比，计算子节点，决定是否复用 Fiber，并标记副作用打上 flags 标记（Placement/Update/Deletion 等）
3. completeWork：向上回溯，收集副作用，创建或更新 DOM 所需的数据，构建DOM实例（但不插入真实DOM）

这一阶段的主要目标是构建 workInProgress tree，并收集需要在 commit 阶段执行的副作用。

由于 render 阶段可能被打断、重试或重复执行，因此这一阶段必须是纯计算过程，不能执行副作用。

> 正因为 render 阶段可能被重复执行，React 将 componentWillMount、componentWillReceiveProps、componentWillUpdate 标记为 UNSAFE，并推荐使用 getDerivedStateFromProps 等纯函数替代。

## 5. commit 阶段

- 不可中断
- 同步执行

主要工作：

1. Before mutation：执行 DOM 变更前的读取逻辑如： getSnapshotBeforeUpdate，
2. Mutation：操作真实 DOM（插入/更新/删除）
3. Layout：执行 useLayoutEffect、componentDidMount、componentDidUpdate
4. Passive Effects: 异步调度并执行 useEffect

commit 阶段之所以不能中断，是因为它会直接操作真实 DOM，必须保证视图更新的一致性。

## 6. 为什么 React 能做到批量更新和局部更新

- **批量更新**：多个 Update 先入队，再统一调度，减少重复渲染
  - React 18 的变化：React 18 之后支持更广泛的**自动批处理**。 不仅 React 事件中的更新会批处理，在 `setTimeout`、`Promise`、原生事件等场景中的更新也会自动合并。
- **局部更新**：React 不会每次都遍历整棵树，而是会通过 `lanes`、`childLanes`、`flags` 和 `bailout` 机制，只处理真正需要更新的子树
  - lanes / childLanes：用于表示当前节点或子树上是否存在待处理更新，以及这些更新的优先级。如果某个子树的 childLanes 不包含本次更新的 lane，直接跳过
  - bailout：如果某个 Fiber 及其子树没有需要处理的更新，React 可以直接复用旧结果，跳过重新计算。 beginWork 中判断 oldProps === newProps && !hasContextChanged() && 无 pending update，则复用整个子树
  - flags / subtreeFlags：会在 render 阶段被收集并向上冒泡，commit 阶段据此快速定位需要执行副作用的节点。

## 7. 一句话总结

React 的更新过程本质上就是：

**把更新封装成 Update 放进 Fiber 队列，通过调度器按优先级安排执行，在 render 阶段计算新 Fiber 树，在 commit 阶段同步更新 DOM。**
