# 代码规范自动化

## Husky

Husky 是一个 Git 钩子管理工具。

Git 在执行某些操作（比如 commit, push）之前或者之后，允许执行一些脚本。
但是 Git 默认的钩子文件在 .git/hooks 目录下，这个目录不会被提交到远程仓库。

Husky 的作用就是让我们能够在如 package.json 之类的地方管理钩子，并确保团队中的成员在安装依赖后，本地的 Git 钩子能够正常工作。

原理：

1. 初始化：当你运行 husky install（现代版本）时，它会在项目根目录创建一个 .husky/ 目录。
1. 配置：它通过修改 Git 的配置 core.hooksPath，将 Git 钩子的搜索路径从默认的 .git/hooks 指向 .husky。
1. 触发：当你执行 git commit 时，Git 会去查找 pre-commit 钩子。因为路径被 Husky 改了，Git 就会执行 .husky/pre-commit 里的脚本。

常用的一些 hooks

- `pre-commit` 当执行 git commit 命令时触发。
- `commit-msg` 对提交消息的格式做自定义的验证。当执行 git commit 命令时触发。

新建一个钩子函数，并写入写入内容：

```bash
echo "npx lint-staged" > .husky/pre-commit

```

## lint-staged

lint-staged 是一个 Git 钩子工具，**它只对 Git 暂存区的内容进行操作**；
并且只能在 git commit 前使用，之后使用无效；需要手动执行`npx lint-staged` **无法自动运行**。

如果没有这个工具，你可能需要在 pre-commit 钩子中运行 npm run lint。会全量检查所有文件会非常慢（可能需要几分钟）。

lint-staged 解决了两个痛点：

- 速度快：只检查你本次修改的文件，不检查没动过的文件。
- 避免污染：防止你在提交 A 功能时，因为项目里其他陈旧代码不符合规范而导致提交失败。

```bash
pnpm add lint-staged -D
```

## commitlint

对 commit-msg 进行校验。

相关的包：

```bash
pnpm add @commitlint/config-conventional @commitlint/cli -D
```

## editorconfig

editorconfig 是给编辑器用的，里面可以定义缩进等，保证在每个编辑器的代码风格是统一的。
